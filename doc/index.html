States

There are three states whereas one state is only triggered by the system.

working: set this state with assume_working broken: set this state with assume_broken pending: this state is assigned to a ressource if the last result is different from the current one.

If a working ressource fails in a test it'll fall into the pending state. This will only change if you either assume your ressource working or broken otherwise it will remain in the state pending forever. If you don't define any possible/reachable temperature rule which assumes your ressource either working or broken it will never change from pending anyway. This is because the initial state for every ressource on application startup is pending. Gotcha: NEVER define your assume_* declarations in an on() rule! Let's say you have on(5) { assume_broken } and a broken ressource got a passed test. The resource is flagged as pending by the system but your on() rule won't be triggered again because it was already called for this incident! An incident is closed as soon as the ressource reaches the state "working" again. Gotcha: You shouldn't declare assume_* declarations in conjunction with nerf. This may lead to unexcpected behaviours.

States: - assume_working - pending - assume_broken
